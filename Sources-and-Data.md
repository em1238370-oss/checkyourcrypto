# Sources and Data — источники и данные (технический документ)

Документ описывает, откуда берутся данные для сайта [crypto-academy-pro.onrender.com](https://crypto-academy-pro.onrender.com/#feature1) с точки зрения фронтенда, бэкенда и внешних сервисов.

Согласован с официальным отчётом **«СКАМ‑МОНИТОРИНГ | Source & Data»** (PDF): методология разделов 2–7 ниже соответствует тому документу.

---

## 0. Источники данных для блока «Результаты в цифрах» (по отчёту Sources and Data)

Цифры на сайте (новые скам‑каналы, потери ₽, число каналов, курсы, Топ‑3) должны браться **только** из перечисленных ниже источников.

### 0.1 TGStat (поиск каналов по крипто/сигналам)

- **Ссылка:** https://tgstat.ru/search?query=криптовалюта&sort=date  
- **API:** https://tgstat.ru/docs/ru/api  
- **Условия отбора:** канал создан менее 14 дней назад; есть VIP/платные услуги от 10 000₽; рост подписчиков за сутки более 500.

### 0.2 Telega (каталог крипто‑каналов)

- **Ссылка:** https://telega.in/catalog?category=cryptocurrencies  
- **Условия:** категория «cryptocurrencies» / «trading signals»; в каталоге менее 14 дней; реклама или VIP от 10 000₽.

### 0.3 Telegram‑чаты с жалобами

- Чаты задаются в настройках (например, `KRO_SOURCE_CHANNELS`). Учитываются сообщения за последние **12 часов**.  
- **Условия:** упоминание канала/сайта (ник или ссылка); указана сумма потерь; по объекту ≥ 2 разных человека.

### 0.4 Итоги для сайта (раздел 6 отчёта)

Переменные, которые формируются отчётом и должны подставляться на сайт:

| Переменная | Описание | Пример на сайте |
|------------|----------|-----------------|
| `new_scam_channels` | Новых скам‑каналов за 12 ч | «47 новых скам-каналов за последние 12 ч» |
| `losses_12h` | Потери ₽ за 12 ч | «12 847 300 Потеряно ₽» |
| `telegram_channels` | Всего Telegram‑каналов в зоне риска | «37 Telegram каналов» |
| `courses_products` | Курсы/фейк‑продукты | «10 Курсы фейк-продуктов» |
| **Топ‑3** | Топ‑3 по потерям за период | Блок «Топ-3 сегодня» |

Обновление: **каждые 12 ч** (LIVE). Чек‑лист отчёта (раздел 7): ссылки в таблице рабочие, жалобы за период, признаки риска и статус заполнены, Топ‑3 из основной таблицы.

### 1.2 Блок «Результаты в цифрах» (на продакшене)

На живой версии сайта есть блок с метриками:
- «47 новых скам-каналов за последние 12 ч»
- «12 847 300 Потеряно ₽»
- «37 Telegram каналов», «10 Курсы фейк-продуктов»
- «LIVE · Обновление каждые 12 ч»

**В текущем репозитории:** этого блока в `index.html` **нет**. То есть либо:
- он добавлен только в деплое (другая ветка/сборка),  
- либо данные планируется подтягивать с API позже.

**Рекомендация:** завести отдельный API (или JSON/конфиг) для этих метрик и подставлять их в разметку или через JS, чтобы не хранить «живые» цифры в HTML.

---

### 1.3 Формы: «Проверить канал» и «Сообщить о разводе»

**Где:** `index.html` — инпуты и кнопки (`.checker-input`, `.checker-file`, `.report-form`).

**Источник данных:** пользовательский ввод. В текущем коде **нет привязки к API**: нет `fetch`/обработчиков для:
- проверки канала по @username / ссылке,
- отправки репорта (канал, сумма ₽, «от кого»).

**То есть:** данные с форм никуда не отправляются, нужна доработка бэкенда и скриптов.

## 2. Чат (Crypto AI Assistant)

### 2.1 Статус доступа (подписка / триал)

**Источник:** Backend API.

**Эндпоинт:** `GET /api/status?userId=...`  
**Где используется:** `scripts/chatbot.js` — функция `fetchStatus(userId)`.

**Данные:** информация о бесплатном периоде и подписке (например, `hasAccess`, `freeTrialEndsAt`). Base URL для API задаётся в скрипте (например, от `window` или конфига).

---

### 2.2 Ответы чата (Mistral AI)

**Источник:** Backend проксирует запросы к **Mistral AI**.

**Эндпоинт:** `POST /api/chat`  
**Тело:** `{ userId, question }`.  
**Ответ:** ответ ассистента (например, `{ answer }` или `{ error }`).

**Где в коде:** `scripts/chatbot.js` — `requestChatResponse(userId, question)`.

Ключ Mistral задаётся в `backend/.env`: `MISTRAL_API_KEY`.

---

### 2.3 Источники под ответами чата (Sources)

**Источник данных:** локальный объект в фронтенде.

**Где в коде:** `scripts/chatbot.js`:
- объект **`cryptoSources`** (строки ~640–701) — темы (bitcoin, ethereum, defi, nft, wallet, scam, dao, gas и т.д.) и для каждой массив ссылок `{ url, title }`;
- **`getSourcesForQuestion(question)`** — по ключевым словам в вопросе выбирает тему и возвращает до 2 ссылок;
- при отрисовке ответа бота под сообщением добавляется блок `.message-sources` с этими ссылками.

**Примеры источников:** bitcoin.org, ethereum.org, Investopedia, Binance Academy, CoinMarketCap, CoinGecko, FTC (скамы) и т.д. — всё зашито в `cryptoSources`, без запросов к API.

---

### 2.4 Fallback-ответы (без Mistral)

**Источник данных:** локальный объект **`cryptoKnowledge`** в `scripts/chatbot.js` (в т.ч. ключевые слова и готовые ответы по темам).

Используется, когда бэкенд не вернул ответ или запрос не удался. Данные статичные, не с сервера.

## 3. Оплата подписки

### 3.1 Создание счёта (крипто)

**Источник:** Backend + внешний платёжный провайдер.

В `SETUP_INSTRUCTIONS.md` указаны **CryptoCloud** (`CRYPTOCLOUD_API_KEY`, `CRYPTOCLOUD_SHOP_ID`). В коде чата вызывается создание инвойса (например, `paymentUrl` или `address`), данные приходят в ответе API и показываются в модалке оплаты.

**Где в коде:** `scripts/chatbot.js` — обработка кнопки оплаты, `fetch` на эндпоинт создания инвойса, отображение суммы, адреса, ссылки на оплату.

---

### 3.2 Карты (CloudPayments)

В инструкции указано «CloudPayments пока не настроено». В коде есть ветка под карты: `data.publicId` и виджет CloudPayments. Данные для виджета приходят с бэкенда (например, `publicId`, `amount`, `invoiceId`, `orderId`).

## 4. Backend (текущая структура по репозиторию)

- В **`crypto-website/backend/`** в репозитории видны: `package.json`, `README.md`, `.env.example`, `node_modules`. Файла **`server.js`** в папке `backend/` в списке файлов не было — возможно, он в другой ветке или деплой идёт с корня.
- В **корне** `crypto-website` лежит **`server.js`** — это другой сервис: портфель, алерты по цене, **CoinGecko** (цены), **Resend** (email). Он не реализует `/api/status`, `/api/chat` и платежи из SETUP_INSTRUCTIONS.

Итого по данным бэкенда чата/подписки:
- **Mistral AI** — ответы чата.
- **CryptoCloud** — крипто-платежи (по инструкции).
- **CloudPayments** — карты (зарезервировано, не настроено).
- Статус подписки/триала хранится/считается на бэкенде (логика должна быть в том сервере, который отдаёт `/api/status` и обрабатывает коллбеки платежей).

## 6. Рекомендации для развития

1. **ТОП-3 и «Результаты в цифрах»:** вынести метрики в API или JSON, подгружать на фронте (или при сборке), обновлять по расписанию (например, «каждые 12 ч»).
2. **Проверка канала и репорты:** добавить эндпоинты на бэкенде и подключить формы в `index.html` к ним (отправка и, при необходимости, отображение результата).
3. **Sources для чата:** при желании можно перенести `cryptoSources` на бэкенд или в конфиг, чтобы менять ссылки без правки JS.
4. **Один backend:** убедиться, что деплой использует тот сервер, где реализованы `/api/status`, `/api/chat` и платежи (CryptoCloud/CloudPayments), а не только корневой `server.js` с портфелем и CoinGecko.

## Спецификация формата отчёта «СКАМ‑МОНИТОРИНГ» (ТЗ)

### 1. Структура документа по дню

1. Для **каждого дня** создаётся один лог‑блок:
   - `Период: DD.MM.YYYY 00:00 – DD.MM.YYYY 23:55 (MSK)`
   - `Время формирования: DD месяц, 23:55 MSK`

2. Ниже идёт раздел:
   - `Временная линия дня:`
   - и строки событий по времени.

3. Внутри одного дня:
   - время идёт по возрастанию от 00:00 до 23:55;
   - **каждая отметка времени (HH:MM) используется не больше ОДНОГО раза**;
   - запрещено копировать один и тот же блок событий (например, 03:00 с теми же каналами) многократно.

4. Списки каналов **не пишутся** в лог по времени. Все каналы и детали по ним фиксируются только в таблице.

---

### 2. Формат временной линии

Формат строки:

`HH:MM — краткое описание события (ОДНА строка)`

Пример:

```
Период: 07.03.2026 00:00 – 07.03.2026 23:55 (MSK)
Время формирования: 7 марта, 23:55 MSK

Временная линия дня:
00:00 — Начат сбор данных за период 00:00–23:55.
00:05 — TGStat/Telega: найдено каналов по фильтрам — TGStat 0, Telega 0.
00:10 — За последние 30 минут новых объектов по фильтрам не найдено, источники доступны.
00:20 — TGStat/Telega: найдено каналов по фильтрам — TGStat 0, Telega 53.
...
23:55 — Завершён сбор данных за день, таблицы обновлены.
```

Правила:
- лог отвечает только на «когда» и «что в общем произошло» (нашли/не нашли, сколько, был ли перерасчёт);
- **никаких списков @каналов и «(3/6 рисков)» в логах**.

---

### 3. Главная таблица «Объекты и риск»

Одна таблица на день, создаётся нормальной вставкой таблицы (не текстом).

#### 3.1. Столбцы (8 штук)

Слева направо:

| № | Столбец | Описание |
|---|---------|----------|
| 1 | Объект / ссылка | Канал: @username + гиперссылка https://t.me/username. Сайт: example.com + ссылка. Если ссылка не открывается или объект не про крипту/сигналы — строку не создавать. |
| 2 | Тип | Одно значение: сигнал‑канал, крипто‑канал (новости), бот, сайт/курс, обучение. Без описаний. |
| 3 | Источник | каталог Telegram, поиск в Telegram, чат с жалобами, поиск в интернете. До двух через «+». |
| 4 | Краткое описание | 1–2 предложения по факту. Только то, что видно в описании и постах. Без оценок «мошенники», «плохой». |
| 5 | VIP / деньги | Как объект зарабатывает. Только реальные данные. «Платных услуг не обнаружено» — если нет. |
| 6 | Жалобы / отзывы | Итог по жалобам. Если сумм нет — не придумывать. «Жалоб за период не найдено» — если нет. |
| 7 | Признаки риска (флаги) | Конкретные флаги по факту: новый канал (< 14 дней), дорогой VIP (от ~10 000₽), обещания x2/x5, только прибыльные сделки и т.д. Через запятую или с новой строки. Ни одного флага нельзя придумать. |
| 8 | Итоговый статус | Резюме по 4–7: в риске (структура + поведение + жалобы), поведенческий риск (жалоб нет), без риска (новости/аналитика), под наблюдением (мало данных). Только из предыдущих столбцов. |

1‑я строка таблицы = заголовки (жирным). Дальше по одной строке на каждый реальный объект.

---

### 4. Приоритет: сигнальные каналы

1. Главный приоритет поиска — **сигнал‑каналы** по крипте.
2. Новости/аналитика, которые не продают сигналы/доход, не считать рисковыми; их можно не включать или помечать «без риска (новости/аналитика)».
3. Для каждого сигнального канала в таблице обязательно: реальная кликабельная ссылка; описание, что делает; как продаёт VIP/курсы; флаги поведения; честная информация по жалобам; итоговый статус.

---

### 5. Связь документа с сайтом

1. Все цифры и объекты, выводимые на сайт (таблицы и метрики), должны быть прослеживаемы до таблицы «Объекты и риск» и при необходимости до таблиц сигналов/VWAP.
2. В конце блока за день — раздел: сколько строк со статусом «в риске»; суммарные зафиксированные потери (если есть); какие объекты в топ‑3 (по жалобам/флагам/потерям).
3. Никаких чисел «из воздуха» — всё только из таблиц.

---

### 6. Жёсткие запреты (для Cursor)

- Запрещено дублировать один и тот же блок логов (одно и то же время и те же каналы) внутри дня.
- Запрещено выводить длинные списки @каналов в логах по времени — все каналы только в таблице.
- Запрещено придумывать каналы, суммы, жалобы и признаки риска.
- Разрешено писать «жалоб нет», «платных услуг не обнаружено», «мало данных», но **нельзя** рисовать картинку «для красоты».
- Если ссылка не открывается — объект не попадает в таблицу вообще.
